# ZTE MC801A API Research

## Overview

The ZTE MC801A (and the broader ZTE CPE family) exposes a local HTTP+JSON API that the built-in web UI uses internally. It is not publicly documented by ZTE, but it has been reverse-engineered by the community and is well-understood.

All communication happens over plain HTTP to `http://192.168.100.1`. There is no HTTPS on the LAN interface.

---

## Authentication

### Flow

1. **Fetch the Login Key (LD)**

   ```
   GET /goform/goform_get_cmd_process?cmd=LD
   ```

   Response:
   ```json
   { "LD": "A3F7..." }
   ```

   `LD` is a random nonce generated by the modem per session.

2. **Compute the password hash**

   ```
   password_hash = SHA256( SHA256(plain_password).upper() + LD )
   ```

   Both operations produce lowercase hex. The inner SHA256 digest is uppercased before concatenation.

   > **Firmware note:** This SHA-256 scheme was confirmed by HAR capture on firmware shipped with the MC801A as of 2026. Older ZTE CPE devices (MF286, MF289) use an MD5-based scheme: `MD5(MD5(pwd).upper() + LD)`. The scraper tries the SHA-256 variant first, then falls back to MD5 variants for compatibility.

3. **POST the login**

   ```
   POST /goform/goform_set_cmd_process
   Content-Type: application/x-www-form-urlencoded
   Referer: http://192.168.100.1/index.html

   isTest=false&goformId=LOGIN&password=<hash>
   ```

   Successful response:
   ```json
   { "result": "0" }
   ```

   A session cookie (`stok` or similar) is set in the response and must be included in all subsequent requests.

### Notes

- Some data fields appear to be readable **without authentication** — particularly read-only network status fields. The PoC should test unauthenticated access first to minimise complexity.
- Some firmware versions enforce stricter session handling. If unauthenticated reads stop working, the full login flow is required.
- The `Referer` header is checked by the modem and **must** be set to `http://192.168.100.1/index.html` (or the main page URL) on all requests, otherwise the modem returns an error or empty response.

## Reading Data

### Endpoint

```
GET /goform/goform_get_cmd_process?multi_data=1&cmd=<comma-separated-fields>
Referer: http://192.168.100.1/index.html
```

The response is a flat JSON object with field names as keys.

### Relevant Signal Fields

#### 5G NR

> **Firmware note:** Field names confirmed against `MC801A_Elisa3_B22`. The `Z5g_*` prefix is mixed-case inconsistently; `Z5g_rsrp` is lowercase while `Z5g_SINR`/`Z5g_RSRQ` are uppercase. `nr5g_*` fields provide band, channel, and PCI.

| Field | Description | Unit | Good range |
| --- | --- | --- | --- |
| `Z5g_rsrp` | 5G NR RSRP | dBm | > −110 dBm |
| `Z5g_RSRQ` | 5G NR RSRQ | dB | > −15 dB |
| `Z5g_SINR` | 5G NR SINR | dB | > 10 dB |
| `nr5g_action_band` | Active 5G NR band (`n78`, `n28`, etc.) | — | — |
| `nr5g_action_channel` | Active NR-ARFCN | — | — |
| `nr5g_pci` | 5G Physical Cell ID (hex) | 0–1007 | — |

#### LTE (anchor cell in NSA mode or fallback)

> **Firmware note:** Field names confirmed against `MC801A_Elisa3_B22`. Older ZTE CPE firmware used bare names (`rsrp`, `rsrq`, `sinr`, `rssi`) without the `lte_` prefix — these return empty on this firmware.

| Field | Description | Unit | Good range |
|---|---|---|---|
| `lte_rssi` | LTE Received Signal Strength Indicator | dBm | > −95 dBm |
| `lte_rsrp` | LTE RSRP | dBm | > −110 dBm |
| `lte_rsrq` | LTE RSRQ | dB | > −15 dB |
| `lte_snr` | LTE SNR (Signal-to-Noise Ratio) | dB | > 5 dB |
| `lte_ca_pcell_band` | LTE primary cell band number | `20`, `3`, etc. | — |
| `lte_ca_pcell_bandwidth` | LTE primary cell bandwidth | MHz | — |
| `lte_ca_scell_band` | First secondary CA cell band | — | — |
| `lte_ca_scell_bandwidth` | First secondary CA cell bandwidth | MHz | — |
| `lte_multi_ca_scell_info` | All CA secondary cell info (CSV) | — | — |
| `lte_pci` | LTE Physical Cell ID (hex) | 0–503 | — |
| `wan_active_band` | Active band string e.g. `LTE BAND 20` | — | — |
| `wan_active_channel` | Active EARFCN | — | — |
| `wan_lte_ca` | CA status (`ca_deactivated`, `ca_activated`) | — | — |

#### Connection and Network State

| Field | Description |
| --- | --- |
| `network_type` | `ENDC` (5G NSA), `NR5G` (5G SA), `LTE`, `WCDMA`, etc. |
| `ppp_status` | `ppp_connected`, `ppp_disconnected` |
| `network_provider` | Operator name e.g. `Telia FI` |
| `signalbar` | Modem's internal signal bar count (0–5) |
| `wan_ipaddr` | Current WAN IPv4 address |
| `ipv6_wan_ipaddr` | Current WAN IPv6 address |
| `cell_id` | Serving LTE Cell ID (hex) |
| `tac` | Tracking Area Code (decimal). OpenCelliD uses this as the "LAC" field in its LTE cell search. |
| `realtime_rx_thrpt` | Current download throughput (bytes/s) |
| `realtime_tx_thrpt` | Current upload throughput (bytes/s) |
| `realtime_rx_bytes` | Session total bytes received |
| `realtime_tx_bytes` | Session total bytes sent |
| `realtime_time` | Session uptime in seconds |
| `pm_sensor_mdm` | LTE modem temperature (°C) |
| `pm_modem_5g` | 5G modem temperature (°C) |

### Example Request

```
GET /goform/goform_get_cmd_process?multi_data=1&cmd=network_type,ppp_status,network_provider,lte_rsrp,lte_rsrq,lte_snr,Z5g_rsrp,Z5g_SINR,nr5g_action_band,wan_ipaddr
```

### Example Response (MC801A_Elisa3_B22)

```json
{
  "network_type": "ENDC",
  "ppp_status": "ppp_connected",
  "network_provider": "Telia FI",
  "lte_rsrp": "-87",
  "lte_rsrq": "-12",
  "lte_snr": "3.6",
  "Z5g_rsrp": "-106",
  "Z5g_SINR": "14.0",
  "nr5g_action_band": "n78",
  "lte_band": "B3",
  "wan_ipaddr": "100.x.x.x"
}
```

Values are returned as **strings**, not numbers. The scraper must parse them.

## Cell ID Decoding

The `cell_id` field returned by the modem is a **hex-encoded E-UTRAN Cell Identifier (ECI)**, a 28-bit value defined in 3GPP TS 36.413.

### ECI Structure

```
 ┌────────────────────────────┬────────────┐
 │       eNB ID (20 bits)     │  Sector    │
 │                            │  (8 bits)  │
 └────────────────────────────┴────────────┘
 Bit 27                   Bit 8  Bit 7  Bit 0
```

| Component | Bits | Formula | Identifies |
|---|---|---|---|
| eNB ID | 27–8 (upper 20 bits) | `ECI >> 8` | Physical base station |
| Sector | 7–0 (lower 8 bits) | `ECI & 0xFF` | Antenna / cell on that tower |

### Decoding Example

Modem reports: `cell_id = "239550c"`

```
ECI (decimal) = 0x239550C = 37,384,460
eNB ID        = 37,384,460 >> 8  = 145,749   ← identifies the tower
Sector        = 37,384,460 & 0xFF =      12   ← antenna sector on that tower
```

### Tower Cross-Reference

The **eNB ID** is the stable cross-reference key for public tower databases:

| Service | URL pattern |
|---|---|
| OpenCelliD | `https://opencellid.org` → search by MCC, MNC, eNB ID |
| Mozilla Location Services | `https://location.services.mozilla.com` |

> **OpenCelliD search requires four fields:** MCC, MNC, LAC, and Cell ID.
> For LTE, **LAC = TAC** (Tracking Area Code) — the modem exposes this as the `tac` field.
> Use the **eNB ID** as the Cell ID value in the search form.
>
> Example for this installation (Telia Finland):
> - MCC: `244`, MNC: `91`
> - LAC (= TAC): value from `tac` modem field
> - Cell ID: `145749` (eNB ID)

For Telia Finland (MCC=244, MNC=91), search for eNB ID `145749` to locate the serving tower on a map and compare against the operator's published coverage data.

### Prometheus Metrics

The exporter exposes a `cellsentry_cell_decoded_info` gauge (always 1) that carries all decoded values as labels:

| Label | Content |
|---|---|
| `cell_id_hex` | Raw hex from modem e.g. `239550c` |
| `cell_id_dec` | Decimal ECI e.g. `37384460` |
| `enb_id` | Base station identifier e.g. `145749` |
| `sector` | Sector / antenna index e.g. `12` |
| `pci_lte_dec` | LTE Physical Cell ID in decimal e.g. `486` |
| `earfcn` | LTE EARFCN (channel frequency) |
| `tac` | Tracking Area Code (= LAC in OpenCelliD) |

## Known Quirks and Failure Modes

| Issue | Cause | Mitigation |
| --- | --- | --- |
| Empty/error JSON returned | Missing or wrong `Referer` header | Always set `Referer: http://192.168.100.1/index.html` |
| `"---"` as a field value | Metric not available in current network mode | Treat as `None`/`NaN` in the scraper |
| Fields missing entirely | Firmware version differences | Defensive key lookup with fallbacks |
| Session expires mid-run | The modem times out inactive sessions | Re-authenticate on 401/session-error responses |
| `Z5g_*` fields all `"---"` | Modem is on LTE-only, 5G not connected | Normal; log and continue |

## Discovering Additional Fields

The web UI JavaScript source (loadable from `http://192.168.100.1/js/`) contains all field names used by the interface. Inspecting the browser's Network tab while navigating the modem's signal status page will reveal the exact `cmd` string the UI uses — this is the most reliable way to find new fields on any given firmware version.

---

## Security Notes

- The modem API uses plain HTTP — credentials and session cookies are transmitted unencrypted. This is only acceptable because communication is strictly local (LAN), never routed over the internet.
- The password hash scheme (MD5+nonce) prevents naive replay attacks but MD5 is not a strong hash function. Do not reuse the modem password elsewhere.
- Credentials should be stored in an `.env` file that is **never committed** (already excluded in `.gitignore`).
