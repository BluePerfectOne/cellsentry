# ZTE MC801A API Research

## Overview

The ZTE MC801A (and the broader ZTE CPE family) exposes a local HTTP+JSON API that the built-in web UI uses internally. It is not publicly documented by ZTE, but it has been reverse-engineered by the community and is well-understood.

All communication happens over plain HTTP to `http://192.168.100.1`. There is no HTTPS on the LAN interface.

---

## Authentication

### Flow

1. **Fetch the Login Key (LD)**

   ```
   GET /goform/goform_get_cmd_process?cmd=LD
   ```

   Response:
   ```json
   { "LD": "A3F7..." }
   ```

   `LD` is a random nonce generated by the modem per session.

2. **Compute the password hash**

   ```
   password_hash = SHA256( SHA256(plain_password).upper() + LD )
   ```

   Both operations produce lowercase hex. The inner SHA256 digest is uppercased before concatenation.

   > **Firmware note:** This SHA-256 scheme was confirmed by HAR capture on firmware shipped with the MC801A as of 2026. Older ZTE CPE devices (MF286, MF289) use an MD5-based scheme: `MD5(MD5(pwd).upper() + LD)`. The scraper tries the SHA-256 variant first, then falls back to MD5 variants for compatibility.

3. **POST the login**

   ```
   POST /goform/goform_set_cmd_process
   Content-Type: application/x-www-form-urlencoded
   Referer: http://192.168.100.1/index.html

   isTest=false&goformId=LOGIN&password=<hash>
   ```

   Successful response:
   ```json
   { "result": "0" }
   ```

   A session cookie (`stok` or similar) is set in the response and must be included in all subsequent requests.

### Notes

- Some data fields appear to be readable **without authentication** ŌĆö particularly read-only network status fields. The PoC should test unauthenticated access first to minimise complexity.
- Some firmware versions enforce stricter session handling. If unauthenticated reads stop working, the full login flow is required.
- The `Referer` header is checked by the modem and **must** be set to `http://192.168.100.1/index.html` (or the main page URL) on all requests, otherwise the modem returns an error or empty response.

## Reading Data

### Endpoint

```
GET /goform/goform_get_cmd_process?multi_data=1&cmd=<comma-separated-fields>
Referer: http://192.168.100.1/index.html
```

The response is a flat JSON object with field names as keys.

## Cell ID Decoding

The `cell_id` field returned by the modem is a **hex-encoded E-UTRAN Cell Identifier (ECI)**, a 28-bit value defined in 3GPP TS 36.413.

| Component | Bits | Formula | Identifies |
|---|---|---|---|
| eNB ID | 27ŌĆō8 (upper 20 bits) | `ECI >> 8` | Physical base station |
| Sector | 7ŌĆō0 (lower 8 bits) | `ECI & 0xFF` | Antenna / cell on that tower |

### Decoding Example

Modem reports: `cell_id = "239550c"`

```
ECI (decimal) = 0x239550C = 37,384,460
eNB ID        = 37,384,460 >> 8  = 145,749   ŌåÉ identifies the tower
Sector        = 37,384,460 & 0xFF =      12   ŌåÉ antenna sector on that tower
```

### Tower Cross-Reference

The **eNB ID** combined with **EARFCN** are the practical cross-reference keys for public tower databases.

> **Note (firmware MC801A_Elisa3_B22):** TAC (Tracking Area Code), which OpenCelliD calls "LAC" in its LTE cell search form, is **not exposed** by this firmware through any known API field name (`tac`, `lte_tac`, `wan_lte_tac`, `lac`, etc. all return empty). The OpenCelliD bulk download (CSV) or their API can be searched by eNB ID directly as an alternative.

| Service | How to find the tower |
|---|---|
| OpenCelliD | Bulk CSV download ŌåÆ filter by MCC=244, MNC=91, eNB ID=145749 |
| Mozilla Location Services | Same approach via bulk download |

### Prometheus Metrics

The exporter exposes a `cellsentry_cell_decoded_info` gauge (always 1) that carries all decoded values as labels:

| Label | Content |
| --- | --- |
| `cell_id_hex` | Raw hex from modem e.g. `239550c` |
| `cell_id_dec` | Decimal ECI e.g. `37384460` |
| `enb_id` | Base station identifier e.g. `145749` |
| `sector` | Sector / antenna index e.g. `12` |
| `pci_lte_dec` | LTE Physical Cell ID in decimal e.g. `486` |
| `pci_5g_dec` | 5G NR Physical Cell ID in decimal e.g. `199` |
| `earfcn` | LTE EARFCN (channel frequency) e.g. `6300` (Band 20 downlink) |

> **TAC/LAC not available:** Firmware MC801A_Elisa3_B22 does not expose the Tracking Area Code through any known API field. Use the eNB ID + EARFCN for tower identification via OpenCelliD bulk data.

## Known Quirks and Failure Modes

| Issue | Cause | Mitigation |
| --- | --- | --- |
| Empty/error JSON returned | Missing or wrong `Referer` header | Always set `Referer: http://192.168.100.1/index.html` |
| `"---"` as a field value | Metric not available in current network mode | Treat as `None`/`NaN` in the scraper |
| Fields missing entirely | Firmware version differences | Defensive key lookup with fallbacks |
| Session expires mid-run | The modem times out inactive sessions | Re-authenticate on 401/session-error responses |
| `Z5g_*` fields all `"---"` | Modem is on LTE-only, 5G not connected | Normal; log and continue |

## Discovering Additional Fields

The web UI JavaScript source (loadable from `http://192.168.100.1/js/`) contains all field names used by the interface. Inspecting the browser's Network tab while navigating the modem's signal status page will reveal the exact `cmd` string the UI uses ŌĆö this is the most reliable way to find new fields on any given firmware version.

## Security Notes

- The modem API uses plain HTTP ŌĆö credentials and session cookies are transmitted unencrypted. This is only acceptable because communication is strictly local (LAN), never routed over the internet.
- The password hash scheme (MD5+nonce) prevents naive replay attacks but MD5 is not a strong hash function. Do not reuse the modem password elsewhere.
- Credentials should be stored in an `.env` file that is **never committed** (already excluded in `.gitignore`).
